01Visão e princípios

O backend é uma API REST (JSON) que serve o frontend React/Vite. Ele guarda a verdade dos dados (classes, tarefas, eventos, recorrências) e resolve o que o cliente não deve recalcular sozinho: expansão de eventos recorrentes, derivação do estado PENDENTE e feriados. O frontend mantém estado local intenso (arrasto, hover, seleção) e só persiste mutações relevantes.

1.1 Princípios que orientam o backend

  • A classe é o eixo. Todo evento pertence a uma classe, que define cor (no cliente) e o padrão de rastreamento de conclusão. O backend nunca infere cor; só guarda o hex da classe.
  • Estado de conclusão é derivado, não armazenado cegamente. PENDENTE é uma função de agora > fim + rastreável + ainda AGENDADO. Persistir só os estados que o usuário decide (CONCLUIDO, REMARCADO); calcular PENDENTE na leitura.
  • Remarcar devolve ao Inbox. Não há reagendamento direto: REMARCADO encerra a ocorrência e recria/reabre a Tarefa de origem com status = INBOX. Fluxo único: Inbox → calendário.
  • Recorrência é regra + ocorrências materializáveis. Uma série guarda a regra; cada ocorrência tem estado próprio e pode ter override de horário/status sem afetar a série.
  • Datas timezone-aware. Armazenar sempre com fuso explícito; o cliente converte para exibição. Nunca persistir naive datetimes.
  • Consultivo, nunca bloqueante. O motor da Fase 2 sugere horários; jamais agenda sozinho. Endpoints de sugestão são read-only e idempotentes.

04Modelo de dados (models)

Cinco entidades, espelhando a Especificação §3. IDs em UUID; auditoria (criado_em/atualizado_em) em todas. Todo registro pertence a um usuário (ver §11) — omitido nos trechos por brevidade, mas presente como FK dono.

4.1 Classe

Tipo de atividade. Define cor e padrão de rastreamento. Editável pelo usuário; o app semeia padrões (Aula, Tarefas básicas, Estudar, Prova, Trabalho).

class Classe(models.Model):

4.3 Evento

Item posicionado no calendário. rastrear_conclusao herda da classe ao criar, com override. status só é significativo quando rastrear_conclusao = true — caso contrário fica null.

class Evento(models.Model):
    class Status(models.TextChoices):
        AGENDADO = "AGENDADO"
        CONCLUIDO = "CONCLUIDO"
        REMARCADO = "REMARCADO"
        # PENDENTE é DERIVADO na leitura — não é gravado (ver §05)

    id = models.UUIDField(primary_key=True, default=uuid4, editable=False)
    titulo = models.CharField(max_length=200)
    descricao = models.TextField(blank=True)
    inicio = models.DateTimeField()   # tz-aware
    fim = models.DateTimeField()      # tz-aware
    classe = models.ForeignKey(Classe, on_delete=models.PROTECT, related_name="eventos")
    rastrear_conclusao = models.BooleanField()
    status = models.CharField(max_length=10, choices=Status.choices, null=True, blank=True)
    origem_tarefa = models.ForeignKey(Tarefa, null=True, blank=True,
                                      on_delete=models.SET_NULL, related_name="eventos")
    regra_recorrencia = models.ForeignKey("RegraRecorrencia", null=True, blank=True,
                                          on_delete=models.SET_NULL, related_name="eventos")
    dono = models.ForeignKey(User, on_delete=models.CASCADE, related_name="eventos")
    criado_em = models.DateTimeField(auto_now_add=True)
    atualizado_em = models.DateTimeField(auto_now=True)

    class Meta:
        indexes = [models.Index(fields=["dono", "inicio", "fim"])]  # queries por janela
        constraints = [models.CheckConstraint(check=Q(fim__gt=F("inicio")), name="ck_evento_fim_apos_inicio")]

Atenção ao status. Ao salvar, se rastrear_conclusao = false → forçar status = null. Se true e sem status → AGENDADO. Centralize isso no serializer/serviço, nunca confie no cliente.

4.5 Ocorrência

Materialização de uma data de um evento recorrente. Existe só quando o usuário toca aquela ocorrência (conclui, remarca, pula ou reagenda só ela). Ocorrências não tocadas são virtuais — geradas pela expansão da regra (ver §06), nunca persistidas.

class Ocorrencia(models.Model):
    id = models.UUIDField(primary_key=True, default=uuid4, editable=False)
    evento = models.ForeignKey(Evento, on_delete=models.CASCADE, related_name="ocorrencias")
    data = models.DateField()  # a data específica desta ocorrência
    inicio_override = models.DateTimeField(null=True, blank=True)
    fim_override = models.DateTimeField(null=True, blank=True)
    status_override = models.CharField(max_length=10, null=True, blank=True)  # CONCLUIDO / REMARCADO / PULADO

    class Meta:
        constraints = [models.UniqueConstraint(fields=["evento", "data"], name="uq_ocorrencia_evento_data")]

Evento            1 — 0..1   RegraRecorrencia

05Máquina de estados

Aplica-se somente a eventos com rastrear_conclusao = true. Os demais (Aula, Tarefas básicas, Prova por padrão) têm status = null — só ocupam horário.

AGENDADO ──agora > fim, sem ação──▶ PENDENTE (derivado)
AGENDADO / PENDENTE ──concluir──▶ CONCLUIDO
PENDENTE ──remarcar──▶ REMARCADO ──▶ Tarefa volta ao INBOX

5.1 PENDENTE é calculado, não gravado

Persistir PENDENTE exigiria um job rodando a cada minuto para virar eventos vencidos — frágil e desnecessário. Em vez disso, o estado efetivo é derivado na serialização:

# services/completion.py
def status_efetivo(evento, agora=None) -> str | None:
    if not evento.rastrear_conclusao:
        return None
    if evento.status in (Evento.Status.CONCLUIDO, Evento.Status.REMARCADO):
        return evento.status
    agora = agora or timezone.now()
    if agora > evento.fim:
        return "PENDENTE"
    return Evento.Status.AGENDADO

O serializer expõe status_efetivo como campo read-only. O banco guarda só AGENDADO/CONCLUIDO/REMARCADO. Para recorrentes, a mesma função roda por ocorrência, considerando status_override e fim_override.

5.2 Transições (ações de escrita)

AçãoEndpointEfeito
ConcluirPOST /eventos/{id}/concluirstatus = CONCLUIDO. Se for ocorrência: grava status_override = CONCLUIDO.
RemarcarPOST /eventos/{id}/remarcarstatus = REMARCADO; recria/reabre a Tarefa de origem com status = INBOX. O evento sai do calendário (some das listas ativas).

Regra-chave. Remarcar não abre um seletor de novo horário. Devolve ao Inbox para reentrar no ritual semanal. Se a tarefa de origem foi apagada, recriar uma nova a partir dos dados do evento (título, classe, esforço). Operação transacional (@transaction.atomic).

06Recorrência e ocorrências

Um evento recorrente guarda inicio/fim da primeira ocorrência (define a hora do dia e a duração) e uma RegraRecorrencia. As datas concretas são geradas sob demanda com dateutil.rrule ao consultar uma janela do calendário.

6.1 Expansão da regra → ocorrências virtuais

# services/recurrence.py
from dateutil.rrule import rrule, WEEKLY, MONTHLY

def expandir(evento, janela_inicio, janela_fim, feriados: set[date]):
    regra = evento.regra_recorrencia
    duracao = evento.fim - evento.inicio
    if regra.tipo == "SEMANAL":
        rule = rrule(WEEKLY, dtstart=evento.inicio, until=regra.data_fim or janela_fim,
                     byweekday=regra.dias)
    else:  # MENSAL
        rule = rrule(MONTHLY, dtstart=evento.inicio, until=regra.data_fim or janela_fim,
                     bymonthday=regra.dias)

    for dt in rule.between(janela_inicio, janela_fim, inc=True):
        if regra.ignorar_feriados and dt.date() in feriados:
            continue
        yield montar_ocorrencia(evento, dt, duracao)  # aplica override se existir no banco

montar_ocorrencia procura uma Ocorrencia persistida para aquela data; se existir, aplica inicio_override/fim_override/status_override (incluindo "PULADO", que omite a ocorrência). Caso contrário, devolve a ocorrência virtual herdando horário e status base.

Custo. Expanda sempre dentro de uma janela limitada (a janela visível do calendário, no máximo alguns meses). Nunca expanda séries infinitas. Exija data_fim ou limite por until = janela_fim.

6.2 Editar/concluir uma ocorrência isolada

Concluir, remarcar, pular ou mover uma única ocorrência cria/atualiza uma Ocorrencia com os overrides — sem tocar no evento-série. Editar "toda a série" altera o Evento e/ou a RegraRecorrencia. O MVP precisa apenas de ocorrência isolada e série inteira; "esta e as futuras" fica para depois.

07Feriados (BrasilAPI)

Feriados são marcadores de dia (não eventos) e servem a dois propósitos: exibição no calendário e o filtro ignorar_feriados da recorrência. Buscar no servidor, nunca no navegador (evita CORS, rate limit e divergência).

https://brasilapi.com.br/api/feriados/v1/{ano}
# services/holidays.py
import requests
from django.core.cache import cache

def feriados_do_ano(ano: int) -> set[date]:
    chave = f"feriados:{ano}"
    cached = cache.get(chave)
    if cached is not None:
        return cached
    resp = requests.get(f"https://brasilapi.com.br/api/feriados/v1/{ano}", timeout=5)
    resp.raise_for_status()
    datas = {date.fromisoformat(item["date"]) for item in resp.json()}
    cache.set(chave, datas, timeout=60 * 60 * 24 * 30)  # 30 dias
    return datas
  • Cache agressivo. Feriados nacionais mudam pouco; 30 dias é seguro. Use o cache do Django (Redis em prod, locmem em dev).
  • Degradação graciosa. Se a BrasilAPI falhar, retornar o último cache ou conjunto vazio — nunca derrubar o request do calendário. Logar o erro.
  • Endpoint próprio GET /feriados?ano=2026 repassa o conjunto ao frontend, que pinta o dia (vermelho) sem chamar a API externa.
  • Escopo MVP: feriados nacionais. Estaduais/municipais ficam fora (a BrasilAPI v1 cobre nacionais).

8.6 Convenções de resposta

  • Códigos: 200/201/204 sucesso · 400 validação · 401 sem auth · 403 não é dono · 404 · 409 conflito (ex.: apagar classe em uso).
  • Erros no formato padrão do DRF: { "campo": ["mensagem"] } ou { "detail": "…" }.
  • Escopo por dono: todo queryset filtra dono=request.user. Nunca vazar dados entre usuários.

09Serializers e validações

  • Classe aninhada na leitura, id na escrita. O EventoSerializer expõe classe como objeto (nome+cor) no GET e aceita classe_id no POST/PATCH.
  • Default de rastrear_conclusao. Se ausente no create, herdar de classe.rastreia_conclusao. Validar coerência de status (ver §4.3).
  • fim > inicio validado no serializer e por CheckConstraint no banco (dupla barreira).
  • Cor validada por regex hex ^#[0-9a-fA-F]{6}$.
  • Dias da recorrência validados por tipo: SEMANAL → 0–6; MENSAL → 1–31; lista não vazia.
  • status_efetivo é SerializerMethodField read-only (chama completion.status_efetivo).
  • Campos da Fase 2 (deadline, esforco_estimado, prioridade) aceitos/expostos desde já, mas sem lógica de ranqueamento no MVP.

10Timezone e datas

  • USE_TZ = True. TIME_ZONE = "America/Sao_Paulo" como referência; armazenar em UTC no banco (padrão do Django) e devolver ISO-8601 com offset.
  • O cliente envia datas tz-aware (com offset). Rejeitar (400) datetimes naive.
  • Comparações de pendência (agora > fim) usam django.utils.timezone.now() (aware).
  • Recorrência: a expansão com rrule opera no fuso do evento; converter as ocorrências para aware antes de serializar. Cuidado com transições de horário de verão (hoje inexistente no BR, mas não assuma fuso fixo — use zoneinfo).
  • Feriados comparam por date (sem hora), no fuso local do usuário.

11Auth e permissões

MVP single-tenant por usuário. Auth via JWT (simplejwt): POST /api/v1/auth/token e /refresh. O frontend guarda o access token e o envia em Authorization: Bearer.

  • IsAuthenticated global; IsOwner por objeto (obj.dono == request.user).
  • Querysets sempre filtrados por dono=request.user — defesa em profundidade além da permissão.
  • Seed de classes padrão no signal post_save de User (Aula, Tarefas básicas, Estudar, Prova, Trabalho) com cores da §4.1.
  • CORS restrito ao domínio do frontend (Vercel) via CORS_ALLOWED_ORIGINS.

Escopo. Cadastro/login social, recuperação de senha e multiusuário com compartilhamento estão fora do MVP (Spec §7.3). Basta um fluxo de auth funcional para isolar dados por usuário.

12Fase 2 — motor de sugestões

Não implementar agora; deixar o terreno pronto. Quando chegar a Fase 2 (Spec §8), o motor consome tarefas do Inbox com deadline + esforco_estimado e a capacidade livre da semana, e propõe horários — sem nunca agendar sozinho.

GET/api/v1/sugestoes?semana=2026-06-15
[
  { "tarefa": "1a2b…", "prioridade": "ALTA", "score": 0.87,
    "horario_sugerido": { "inicio": "2026-06-16T14:00-03:00", "fim": "…17:00-03:00" } }
]
  • Score = urgência (decai com distância à deadline) × esforço (precisa começar cedo?) × viabilidade (cabe nas lacunas livres?). Tarefas sem deadline → prioridade baixa. Pesos parametrizáveis.
  • Read-only e idempotente. O endpoint só calcula; aceitar uma sugestão reutiliza POST /tarefas/{id}/promover com o horário escolhido.
  • Lacunas livres = janela do calendário menos eventos existentes (incluindo ocorrências expandidas).

14Testes e qualidade

  • Serviços primeiro: testes unitários de status_efetivo (todas as fronteiras de tempo), expansão de recorrência (semanal/mensal, ignorar feriados, data_fim, override de ocorrência) e transição de remarcar (volta ao Inbox, atômica).
  • API: testes de contrato por endpoint — escopo por dono (não vaza dados), 409 ao apagar classe em uso, validação fim > inicio, rejeição de janela aberta.
  • Factory de dados (factory_boy ou fixtures), pytest-django.
  • Lint/format: ruff + black; checagem de migrations pendentes no CI.
  • Cobrir o caminho do arrasto: promover herda classe e rastreamento corretamente.
